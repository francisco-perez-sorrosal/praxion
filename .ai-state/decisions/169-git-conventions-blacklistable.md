## Context

Open question #5 in the research: "Is `vcs/git-conventions.md` core or blacklistable?" The rule is always-loaded (~3,560 chars). It contains:

- Commit scope discipline (one logical change per commit, etc.)
- Staging discipline (`git add -A` discouraged)
- Secrets-handling rules
- Commit message format (type prefix, subject under 50 chars, imperative mood)
- **"No AI authorship"**: commits do not carry `Co-Authored-By: Claude` lines
- Examples

Most of these are *project-policy* — teams using Conventional Commits, squash-only merges, or different staging conventions are legitimately at odds with Praxion's specific formulation. A Praxion project should be able to opt out of these conventions without losing the behavioral contract.

But ONE clause is genuinely behavioral and load-bearing: "No AI authorship in commit messages." This is a hard property of Praxion-authored work (the human owns the commit; AI is not credited). It should be invariant across all Praxion projects, regardless of their commit-format preferences.

## Decision

1. **`rules/swe/vcs/git-conventions.md` is BLACKLISTABLE** (`core: false`, `install: hook-deliver`). Projects with different VCS conventions can disable it via `.claude/praxion-rules.yaml`.

2. **Before** flipping the install mode for this rule, the "No AI authorship in commit messages" clause is **relocated** to a core, always-loaded surface — specifically `rules/swe/agent-behavioral-contract.md` (which gains a single sentence after the four-behavior list) OR `rules/CLAUDE.md` (in a small "Commit boundary" subsection).

   The relocation target is decided at implementation time based on minimum-edit considerations: `rules/CLAUDE.md` is shorter and a more natural home for a single rule; `agent-behavioral-contract.md` is more semantically aligned. Defer to the implementer; document the choice in `LEARNINGS.md`.

3. The relocated line reads exactly: "No AI authorship in commit messages (`Co-Authored-By: Claude`, `Generated by Claude`, etc.) — the human owns the commit." This text already exists in `git-conventions.md`; the migration is verbatim.

4. After relocation, `git-conventions.md` retains everything else and can be blacklisted by projects that prefer different commit conventions.

## Considered Options

### Option 1 — Keep `git-conventions.md` core (non-disableable)

**Pros:** Single source for all commit discipline.
**Cons:** Forces every Praxion project to adopt Praxion's exact commit format. Inconsistent with the user's framing that the blacklist exists precisely so projects can opt out of rules that don't fit. Most of the rule is project-policy, not behavioral contract. Rejected.

### Option 2 — Make `git-conventions.md` blacklistable; lose "no AI authorship" guarantee for projects that disable it

**Pros:** Minimal change.
**Cons:** Allows a project to silently disable the no-AI-authorship invariant by blacklisting the whole rule. That invariant is genuinely Praxion-wide; it should not be opt-out-able. Rejected.

### Option 3 — Split: relocate the behavioral clause to a core rule, then blacklist the project-policy parts (chosen)

**Pros:** Surgical. Each clause lives at the appropriate tier. Projects gain real choice over commit format; Praxion retains the no-AI-authorship invariant.
**Cons:** Requires touching two rule files. Mitigated by the relocation being a verbatim line copy.

### Option 4 — Create a new `core/git-essentials.md` (or similar) holding the behavioral clauses

**Pros:** Clean separation.
**Cons:** Adds a new always-loaded rule (token cost). Two always-loaded VCS rules is worse than one with a single relocated line. Rejected on the token-budget grounds that drive this whole design.

## Consequences

**Positive:**

- Projects can adopt Conventional Commits, Gitmoji, or any other format by disabling `git-conventions.md`.
- The no-AI-authorship invariant is preserved across all projects regardless of their commit-format preferences.
- Reduced always-loaded burden for projects that don't share Praxion's exact commit format.

**Negative:**

- One additional rule-file edit during implementation (`agent-behavioral-contract.md` gains a line).
- Sentinel BC03 verifies all 11 pipeline agents reference `agent-behavioral-contract.md` — unchanged, still passes.
- Sentinel checks for the no-AI-authorship line may need to be added to verify the relocation succeeded. Defer to verifier/sentinel scope.

**Operational:**

- Implementer adds the line to the chosen core rule.
- Verifier confirms the line exists in the relocation target before any project disables `git-conventions.md`.
- The migration plan in `SYSTEMS_PLAN.md` includes this as a Phase B step (before Phase C flips `install: hook-deliver`).

**Cross-cutting:**

- Depends on the `core: true` invariant from `dec-168` — the relocation target must be a core rule for the invariant to hold.
- Pairs with `dec-167` — the hook-delivery mechanism is what makes the blacklisting effective.
