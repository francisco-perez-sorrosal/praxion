## Context

The systems-architect (`dec-169`) decided to make `rules/swe/vcs/git-conventions.md` blacklistable, and to relocate the "No AI authorship in commit messages" clause to a core, always-loaded rule before the install mode flips. The architect deferred the relocation target choice to implementation time, stating: "Defer to the implementer; document in LEARNINGS.md." The two candidate targets were `rules/swe/agent-behavioral-contract.md` (more semantically aligned) and `rules/CLAUDE.md` (minimum edit).

## Decision

The clause relocates to `rules/swe/agent-behavioral-contract.md`, appended after the existing `Self-test:` line.

The verbatim text is: "No AI authorship in commit messages (`Co-Authored-By`, `Generated by`, etc.) — the human owns the commit."

The clause also remains in `rules/swe/vcs/git-conventions.md` (intentional dual presence) so that projects that do NOT blacklist git-conventions continue to see the full instructional context. The authoritative enforcement source is `agent-behavioral-contract.md`; the git-conventions occurrence is instructional duplication that naturally disappears for projects that blacklist git-conventions.

## Considered Options

### Option 1 — `rules/CLAUDE.md`

**Pros:** Shorter file; minimum edit (one line added to a short file).
**Cons:** `rules/CLAUDE.md` is about "how the Praxion rule system works" — meta-rules. Embedding a behavioral constraint about commit authorship here is a category mismatch. A reader scanning `CLAUDE.md` would not expect to find this constraint there.

### Option 2 — `rules/swe/agent-behavioral-contract.md` (chosen)

**Pros:** The clause is a behavioral invariant (the human owns the commit). The behavioral contract is the correct semantic home for invariants that apply across all Praxion-assisted work. A reader looking for what behaviors Praxion enforces will check this file first. Sentinel BC03 already validates that all agents reference this file — adding the clause here means it is implicitly validated by the sentinel's coverage check.

**Cons:** Adds one line to a file that was originally scoped to the four-behavior list (Surface Assumptions, Register Objection, Stay Surgical, Simplicity First). Minor scope expansion, but well-justified.

### Option 3 — New `rules/swe/core/git-essentials.md`

**Pros:** Clean isolation.
**Cons:** Adds a new always-loaded rule, increasing the token budget. The architect explicitly rejected this option on token-budget grounds in `dec-169`. Rejected here for the same reason.

## Consequences

**Positive:**
- The no-AI-authorship invariant is enforced by a core, always-loaded rule that cannot be blacklisted by any project config.
- Sentinel BC03 (verifying agent-behavioral-contract.md references) implicitly covers this constraint.
- Projects that blacklist `git-conventions.md` still have the no-AI-authorship guarantee enforced.

**Negative:**
- Temporary dual presence of the clause in both files (until a project disables git-conventions.md, at which point the hook-delivery path omits it from that project's session).
- Minor scope expansion of `agent-behavioral-contract.md`.

**Operational:**
- Implementation Step 4 adds the clause to `agent-behavioral-contract.md`.
- Step 4 MUST complete before Step 12 (the manifest install-mode flip).
- The clause is NOT removed from `git-conventions.md`; removal is implicit for blacklisting projects.

**Cross-cutting:**
- Depends on `dec-169` (git-conventions classified blacklistable; relocation mandated).
- Pairs with `dec-168` (core protection) — `agent-behavioral-contract.md` has `core: true`, so the clause is protected.
