# `doc_manifest.yaml` Schema

Reference for `.ai-state/doc_manifest.yaml` — the entry-point spine that the per-project dashboard reads to discover and route every documentation surface in the project. Generated by `scripts/build_doc_manifest.py`; a future doc-manifest-freshness check (tracked as `td-028`) will validate it stays in sync with the filesystem.

Back-link: [`../SKILL.md`](../SKILL.md) · Convention rule: [`../../../rules/writing/html-output-conventions.md`](../../../rules/writing/html-output-conventions.md) (path-scoped to `dashboard_app/**`, `**/doc_manifest.yaml`).

## Purpose

The manifest is **the dashboard's table-of-contents source-of-truth**. The dashboard reads it once at startup via `getDocumentationData()` to build navigation; it does not re-walk the filesystem on every page load. The manifest is a flat YAML list of doc-surface descriptors — type, location, renderer, audience, relationships. Adding a new doc to the project is a matter of running the generator (or letting the post-commit hook run it) and the dashboard picks up the new surface automatically.

## File location and lifecycle

- **Path:** `.ai-state/doc_manifest.yaml` (committed; lives alongside other persistent project intelligence).
- **Producer:** `scripts/build_doc_manifest.py` — walks `docs/`, `.ai-state/`, `.ai-work/<active-slug>/`, and a small set of root-level files (`README.md`, `CLAUDE.md`, `AGENTS.md` if present); emits the YAML.
- **Consumers:** the dashboard's navigation builder (`getDocumentationData()` in `dashboard_app/src/server/view-models/documentation.ts`); `doc-engineer` agent (when validating doc structure); a future doc-manifest-freshness check (`td-028`).
- **Update cadence:** regenerated by `git post-commit` when any tracked `.md`, `.yaml`, `.json`, or `.svg` under doc-relevant paths changes. Manual regen via `python3 scripts/build_doc_manifest.py`.

## Top-level structure

```yaml
schema_version: 1
generated_at: 2026-05-09T12:34:56Z       # ISO 8601 UTC; updated by generator
generator_version: praxion-0.7.0          # bump when generator emits new fields
project_name: Praxion                     # from project root .claude-plugin/plugin.json or CLAUDE.md
project_slug: praxion                      # kebab-case
surfaces:
  - id: docs-getting-started               # stable; kebab-case; <doc-tree>-<basename>
    path: docs/getting-started.md
    type: markdown                         # markdown | yaml | json | svg | html | jupyter
    diataxis: tutorial                     # tutorial | how-to | reference | explanation | concepts | (omit when n/a)
    audience: developer                    # developer | contributor | end-user | architect | agent
    renderer: tutorial_shell               # component name resolved by the dashboard's renderer registry (dashboard_app/src/components/registry.ts) — a Diátaxis-shell key (reference/explanation/default, with tutorial/how-to/concepts aliasing their respective shells) or a content-type key
    share_out: false                       # pre-render to .html for share-out artifacts
    title: "Getting started"               # human-readable; from frontmatter or first H1
    summary: "First-time setup; build a small URL shortener through the full pipeline."
    surfaces_referenced:                   # outbound links to other surface ids in this manifest
      - docs-greenfield-onboarding
      - docs-existing-project-onboarding
      - docs-claude-ecosystem-learning-resources
    diagrams:                              # diagrams embedded by this surface (paths to rendered SVGs)
      - docs/diagrams/getting-started-pipeline/rendered/getting-started-pipeline.svg
    last_modified: 2026-05-08              # mtime as ISO date
    frontmatter:                           # passthrough of source frontmatter for renderer use
      diataxis: tutorial
      audience: developer
groups:                                    # optional: sidebar grouping for the dashboard
  - id: tutorials
    label: Tutorials (learning by doing)
    surface_ids:
      - docs-getting-started
      - docs-greenfield-onboarding
      - docs-existing-project-onboarding
      - docs-ml-training-onramp
  - id: how-to
    label: How-to guides
    surface_ids: [...]
  - id: reference
    label: Reference
    surface_ids: [...]
  - id: explanation
    label: Explanation & concepts
    surface_ids: [...]
  - id: pipeline-state                     # ephemeral .ai-work/<active-slug>/ artifacts
    label: In-flight pipeline
    surface_ids: [...]
    transient: true                        # group disappears when no active pipeline
```

## Field reference (per surface)

| Field | Type | Required | Purpose |
|---|---|---|---|
| `id` | string | yes | Stable kebab-case identifier; `<doc-tree>-<basename>` (e.g., `docs-architecture`, `ai-state-design`, `ai-work-implementation-plan`). Surfaces referenced cross-surface use this id. |
| `path` | string | yes | Project-relative path. Used by the dashboard to read the source. |
| `type` | enum | yes | `markdown` \| `yaml` \| `json` \| `svg` \| `html` \| `jupyter`. Drives the default renderer if `renderer` is omitted. |
| `diataxis` | enum | when applicable | `tutorial` \| `how-to` \| `reference` \| `explanation` \| `concepts`. Read from frontmatter; drives the Diátaxis-typed shell selection. |
| `audience` | enum | recommended | `developer` \| `contributor` \| `end-user` \| `architect` \| `agent`. Read from frontmatter. Drives audience-specific groupings or filters. |
| `renderer` | string | recommended | Component name resolved by the dashboard's renderer registry (`dashboard_app/src/components/registry.ts`) — a Diátaxis-shell key (`reference`/`explanation`/`default`, with `tutorial`/`how-to`/`concepts` aliasing their respective shells) or a content-type key. If omitted, generator picks a default per `type` + `diataxis`. |
| `share_out` | bool | default `false` | When `true`, the pre-commit hook generates a sibling `.html` for sharing. Read from frontmatter. |
| `title` | string | yes | Human-readable; from frontmatter `title:` or first H1. |
| `summary` | string | recommended | One-line description; from frontmatter `summary:` or first non-heading paragraph (truncated). |
| `surfaces_referenced` | list[string] | optional | Other surface ids this doc links to; built by parsing markdown links and resolving them against the manifest. |
| `diagrams` | list[string] | optional | Paths to rendered diagram SVGs/PNGs embedded in the surface. |
| `last_modified` | ISO date | yes | File mtime; used by sentinel to detect drift. |
| `frontmatter` | object | optional | Passthrough of source-file frontmatter (any keys), so renderers can access everything without re-parsing. |

## Reserved renderer names

Initial set, matching the shell registry in `dashboard_app/src/components/registry.ts` and the shells in `dashboard_app/src/components/shells/` (per the HTML output conventions rule):

| Renderer | Default for | Notes |
|---|---|---|
| `tutorial` | `diataxis: tutorial` | Progress-through-steps wrapper (`TutorialShell`) |
| `how-to` | `diataxis: how-to` | Numbered-checklist wrapper (`HowToShell`) |
| `reference` | `diataxis: reference` | Sortable-tables + anchored-nav wrapper (`ReferenceShell`) |
| `explanation` | `diataxis: explanation` | Wide-text + illustration-heavy wrapper (`ExplanationShell`) |
| `concepts` | `diataxis: concepts` | Glossary + conceptual map wrapper (`ConceptsShell`) |
| `markdown` | fallback for any `type: markdown` with no `diataxis:` | Plain markdown render (`DefaultShell`) |

Custom renderers can be added by dropping a new shell component under `dashboard_app/src/components/shells/<name>.tsx` (export it), and registering it in `dashboard_app/src/components/registry.ts`'s `RENDERER_REGISTRY` map (and the `shells/index.ts` barrel if used).

## Generator behavior

`scripts/build_doc_manifest.py` walks the project filesystem and emits the YAML manifest the dashboard reads at startup via `getDocumentationData()` in `dashboard_app/src/server/view-models/documentation.ts`. It covers the following roots in order, emitting one surface entry per qualifying file:

1. **Repo-root surfaces:** `README.md`, `README_DEV.md`, `CLAUDE.md`, `CHANGELOG.md` — fixed list, manually curated in the generator's config block.
2. **`docs/**`** — every `.md` file is a surface. The generator reads frontmatter for `diataxis:` and `audience:`. Inferred `id` from path (e.g., `docs/architecture.md` → `docs-architecture`).
3. **`.ai-state/`** — `DESIGN.md`, `SYSTEM_DEPLOYMENT.md`, `TECH_DEBT_LEDGER.md`, `TECH_DEBT_RESOLVED.md`, `decisions/<NNN>-*.md`, `decisions/DECISIONS_INDEX.md`, `idea_ledgers/IDEA_LEDGER_*.md`, `sentinel_reports/SENTINEL_REPORT_*.md`, `metrics_reports/METRICS_REPORT_*.{md,json}`. Each gets a surface entry.
4. **`.ai-work/<active-slug>/`** — when present, all canonical pipeline documents (`IDEA_PROPOSAL.md`, `RESEARCH_FINDINGS.md`, `SYSTEMS_PLAN.md`, `IMPLEMENTATION_PLAN.md`, `WIP.md`, `LEARNINGS.md`, `VERIFICATION_REPORT.md`, `traceability.yml`, `TEST_RESULTS.md`). Marked with `transient: true` at the group level.

The generator skips:

- Files under `node_modules/`, `.git/`, `.claude/`, `.venv/`, `tmp/`, `dist/`, `build/`.
- Files matching `**/diagrams/**/src/**` — diagram source files are not surfaces; their rendered SVGs are referenced in `diagrams:` lists on the parent surface.
- Files explicitly excluded via a `.docmanifest_ignore` glob in the project root (Claude-Code-style; one pattern per line).

## Planned sentinel check (tracked as `td-028`)

A future doc-manifest-freshness check would validate: manifest exists and parses; `generated_at` ≥ latest mtime of qualifying surfaces; no orphan entries; no missing surfaces; `surfaces_referenced` ids resolve; `diagrams:` paths exist under `**/diagrams/**/rendered/`. The check needs a new sentinel ID (`EC08` or later) — `EC07` is already assigned to the AaC golden-rule check.

## Worktree concurrency

The manifest is regenerated per worktree (each worktree has its own `.ai-state/doc_manifest.yaml`). At merge-to-main, the post-merge hook regenerates the manifest from the merged tree state — there is no merge driver; the generator is deterministic given the filesystem state, so post-merge regen is the canonical reconciliation. Conflicts in the YAML during merge are resolved by re-running the generator (delete the conflict markers, run `python3 scripts/build_doc_manifest.py`, commit the result).

## Migration from the existing implicit structure

Historical note: prior to the Next.js dashboard migration, pages were hardcoded per `streamlit_app/`'s file structure. The active dashboard (`dashboard_app/`) reads the manifest at startup via `getDocumentationData()` and dispatches through the renderer registry. The migration is complete; `streamlit_app/` has been removed (commit `313a50e`).

## Self-test before committing manifest changes

- Did I edit the YAML by hand? **Don't.** Re-run the generator instead.
- Did I add a new doc surface and forget to regenerate? Run `python3 scripts/build_doc_manifest.py`.
- Did I add a new renderer? Drop a new shell under `dashboard_app/src/components/shells/<name>.tsx`, register it in `dashboard_app/src/components/registry.ts`'s `RENDERER_REGISTRY`, and update the "Reserved renderer names" table in this reference.
- Is the manifest up to date? Run `python3 scripts/build_doc_manifest.py` and verify no diff.
