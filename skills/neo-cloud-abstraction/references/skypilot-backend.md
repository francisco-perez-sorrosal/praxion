SkyPilot `~=0.12` (verified 0.12.1 on PyPI, 2026-05-03) covers 20+ cloud providers
(AWS, GCP, Azure, Lambda Labs, RunPod, Crusoe, CoreWeave, Modal, Vast.ai, and more) with
a single YAML task spec. It handles spot recovery, multi-cloud failover, and cost
optimization automatically. Use SkyPilot when exploring providers or running multi-cloud
workflows before committing to a specific provider.

## Configuration

```yaml
# .ai-state/neo_cloud_backend.yaml
backend: skypilot
```

**Install SkyPilot** (project-level; Praxion teaches, not installs):

```bash
pip install "skypilot[all]~=0.12"   # all providers
# or narrow to specific providers:
pip install "skypilot[aws,gcp,lambda]~=0.12"
```

SkyPilot requires cloud credentials in their standard locations:
`~/.aws/`, `~/.gcp/`, `~/.azure/`, or provider-specific config. Never put credentials
in the `training_job_descriptor`.

## Descriptor → SkyPilot YAML Translation

`/run-experiment` translates the `training_job_descriptor` into a SkyPilot task YAML
before calling `sky launch`. The mapping is:

| Descriptor field | SkyPilot YAML path | Notes |
|---|---|---|
| `gpu_type` | `resources.accelerators` | e.g., `H100: 1`; SkyPilot uses `GPU_TYPE: count` dict syntax |
| `gpu_count` | folded into `resources.accelerators` count | |
| `container_image` | `workdir` + `setup` Docker pull, or `image_id` for managed images | |
| `env_vars` | `envs` | dict; SkyPilot passes these to the task |
| `volume_mounts` | `file_mounts` | SkyPilot path → cloud bucket or local path mapping |
| `wall_clock_seconds_max` | `resources.job_recovery.max_restarts_on_errors` + spot timeout | Approximate; SkyPilot timeout granularity is minutes |
| `gpu_hours_budget` | `spot_recovery: fail_fast` + external budget gate in `/run-experiment` | SkyPilot does not directly cap by GPU-hours; `/run-experiment` polls and cancels |
| `artifact_paths` | `file_mounts` reverse direction via `sky rsync-down` after run | |
| `entry_command` | `run` | The shell command to execute on the remote machine |

**Example SkyPilot task YAML** (generated from a descriptor with H100, 1 GPU):

```yaml
# generated by /run-experiment from training_job_descriptor
resources:
  accelerators:
    H100: 1
  use_spot: true    # spot enabled by default in SkyPilot for cost savings

envs:
  LEARNING_RATE: "3e-4"
  BATCH_SIZE: "32"

file_mounts:
  /data: s3://my-bucket/training-data

run: |
  python train.py \
    --lr $LEARNING_RATE \
    --batch-size $BATCH_SIZE \
    --output /workspace/checkpoints
```

## Lifecycle Operations — SkyPilot CLI mapping

| Operation | SkyPilot CLI command | Notes |
|---|---|---|
| `create()` | `sky launch -c <cluster> task.yaml` | Provisions cluster + runs task; returns cluster name as `job_id` |
| `start()` | `sky start <cluster>` | Re-start a stopped cluster; no-op if already running |
| `status()` | `sky status <cluster>` | Parse output for `UP` / `STOPPED` / `INIT` states |
| `log_stream()` | `sky logs <cluster> <job_id>` | Tail logs; blocks until job completes or is cancelled |
| `cancel()` | `sky cancel <cluster> <job_id>` or `sky down <cluster>` | Cancel running job; `sky down` tears down cluster too |
| `artifact_fetch()` | `sky rsync-down <cluster> <remote_path> <local_path>` | Pulls outputs/checkpoints to local |
| `list()` | `sky status --all` | Lists all clusters; parse for project-relevant entries |
| `pricing_query()` | `sky show-gpus <gpu_type>` | Shows per-provider prices for the GPU type; parse lowest-cost option |

**Using the Python API** (alternative to CLI subprocess calls):

```python
import sky

# create + launch
task = sky.Task.from_yaml("task.yaml")
sky.launch(task, cluster_name=run_tag, detach_run=True)

# status
cluster_records = sky.status(cluster_names=[run_tag])
status = cluster_records[0]["status"]   # sky.ClusterStatus enum

# cancel
sky.cancel(cluster_name=run_tag, all=True)

# down
sky.down(cluster_name=run_tag)
```

Prefer the Python API over CLI subprocess calls when implementing `/run-experiment` — it
avoids shell-parsing fragility and integrates cleanly with the descriptor translation.

## Spot Recovery and Multi-Cloud Failover

SkyPilot manages spot preemption automatically when `use_spot: true`:

- On preemption, SkyPilot re-launches on the same or a fallback provider
- Specify fallback order with `resources.ordered_overrides`:

```yaml
resources:
  accelerators:
    H100: 1
  use_spot: true
  ordered_overrides:
    - cloud: lambda
    - cloud: runpod
    - cloud: crusoe
```

- Checkpoint-and-resume is the user's responsibility — SkyPilot restarts from the beginning
  of the `run:` block. Write checkpoints frequently and add resume logic in `train.py`.

## Version Drift Note

SkyPilot follows semantic versioning within the `0.x` series. Schema changes within `0.12.x`
are backward-compatible. Flag for skill refresh when:

- `0.13.0` releases (minor bump — inspect YAML schema for breaking changes)
- SkyPilot Agent Skill API changes (used by `/run-experiment` for Claude Code integration)

Run `pip index versions skypilot` to check the current release before refreshing.

## When to Move from SkyPilot to RunPod Direct

After validating your training pipeline on SkyPilot and deciding to commit to RunPod,
switch to the direct adapter for lower latency and native MCP integration:

```yaml
# .ai-state/neo_cloud_backend.yaml
backend: runpod-direct
```

See [references/runpod-direct-adapter.md](runpod-direct-adapter.md) for the migration recipe.
